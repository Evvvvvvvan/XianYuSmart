# XianYuSmart 多租户商家运营平台设计

## 目标

在现有账号、自动回复、自动发货、在线聊天和订单能力上，补齐多租户隔离、商品自动评价、商品自动擦亮、发布资料、商品采集分销、通知风控和返佣运营闭环。所有自动动作默认关闭，仅对显式启用的商品或规则生效。

## 核心原则

- 租户以系统用户 ID 标识，HTTP 请求由认证上下文提供租户 ID。
- 业务表统一增加 `tenant_id`，查询层自动附加租户条件；后台任务按账号归属处理。
- 商品自动评价和自动擦亮使用两个独立开关，关闭后不产生平台请求。
- 运营类配置统一使用“资源”模型，执行过程统一使用“任务”模型，减少重复表和重复代码。
- 平台操作串行限速、失败留痕、凭证失效立即停止，避免无效重试放大风险。
- 外部项目仅用于核对业务流程，所有实现保持当前项目命名、结构和代码风格。

## 数据模型

### 多租户

- 现有业务表增加 `tenant_id` 和组合索引。
- `xianyu_account` 的账号唯一约束调整为租户内唯一。
- 与账号直接关联的数据在写入时由数据库按账号归属校正租户 ID，支持 WebSocket 与定时任务线程。
- 间接关联数据按卡密配置或关键词规则归属校正租户 ID。

### 商品自动化

`xianyu_goods_config` 增加：

- `xianyu_auto_rate_on`：自动评价开关，默认 0。
- `xianyu_auto_polish_on`：自动擦亮开关，默认 0。
- `last_polish_time`：最近成功擦亮时间，用于每日幂等。

### 运营资源

`merchant_resource` 按 `resource_type` 承载：

- `ADDRESS`：地址库。
- `MATERIAL`：素材库，包含标题、图片、详情、淘口令、短链、库存和发布状态。
- `SUPPLY`：采集货源。
- `PROMOTION_ACCOUNT`：返佣账号。
- `SELECTION_RULE`：选品规则。
- `PUBLISH_RULE`：发布规则。
- `DELETE_RULE`：删除规则。
- `ANNOUNCEMENT`：公告。
- `FEEDBACK`：系统反馈。
- `RISK_EVENT`：风控事件。

通用字段保存名称、状态、关联账号、远端商品、库存、金额、计划时间和 JSON 扩展信息。

### 任务与分销

- `merchant_task`：采集、发布、删除、补偿、评价、擦亮任务及执行日志。
- `merchant_distribution`：货源、素材、发布商品、佣金、结算状态和对接记录。

## 业务流程

### 自动评价

1. 仅查询存在自动评价商品的账号。
2. 拉取待评价订单。
3. 按订单 ID 匹配本地订单并定位商品。
4. 再次校验该商品开关，未匹配或未开启直接跳过。
5. 串行提交评价，成功或失败均写入任务与操作日志。

### 自动擦亮

1. 查询开启自动擦亮且当天未成功执行的在售商品。
2. 按账号和商品串行调用擦亮接口。
3. 成功或平台提示当日已擦亮时更新 `last_polish_time`。
4. 凭证失效或风控响应时停止该账号当轮任务。

### 素材、发布与返佣

1. 商品采集或手工录入形成货源资源。
2. 选品规则筛选候选货源并生成素材。
3. 发布规则按时间创建发布任务，发布结果回写远端商品 ID。
4. 删除规则创建删除任务；失败任务进入补偿队列。
5. 分销记录关联货源、素材、账号、远端商品与佣金，结算状态可追踪。

## 前端

- 商品管理保留现有视觉体系，增加“自动评价”“自动擦亮”两个紧凑开关。
- 新增“运营中心”，按资料发布、采集分销、返佣规则、任务日志、通知风控分组展示。
- 表单仅呈现当前资源所需字段，统一列表、状态、立即执行和删除交互。
- 页面不展示其他租户数据，不暴露服务器、Cookie 或内部错误堆栈。

## 性能与可靠性

- 所有租户、账号、状态和到期任务查询建立组合索引。
- 调度使用固定批量、数据库状态抢占和有限重试。
- 擦亮按日幂等，评价以平台待评价列表和本地订单关联共同去重。
- 单账号串行、账号间有限并发，避免连接和平台请求峰值。
