# XianYuSmart 商家履约闭环优化设计

## 1. 目标

本轮优化以单商家虚拟商品和卡密经营为核心，优先缩短异常订单发现与处理路径，降低漏单、重复发货和库存不足带来的经营风险。

核心目标：

1. 首页待办可直接定位对应订单或卡密仓库。
2. 发货记录准确展示任务状态和失败原因。
3. 明确失败的订单可安全重新排队。
4. 人工核对订单禁止自动重发，避免重复交付卡密。
5. 列表查询和首页刷新设置资源边界，保持高并发下的低占用。
6. 保持现有视觉风格，仅增加必要筛选和操作，不增加装饰性组件。
7. 商品管理支持安全维护本地运营资料，并提供平台修改与同步回读闭环。

## 2. 当前问题

### 2.1 首页待办缺少处理闭环

等待履约、人工核对和履约失败均跳转到通用发货记录页，进入页面后仍需人工查找目标订单。卡密库存预警只进入卡密管理页，无法直接定位低库存仓库。

### 2.2 订单状态信息丢失

数据库已保存 `delivery_status`、`fail_reason` 和 `last_error_message`，发货记录接口只返回传统 `state` 字段，页面只能显示待发货、已发货或失败，无法区分处理中、等待重试和人工核对。

### 2.3 失败订单缺少安全恢复入口

任务服务已具备重新排队能力，但管理页面没有对应入口。异常处理依赖后台自动重试或人工触发其他流程，处理路径不直观。

### 2.4 查询资源缺少边界

发货记录的页码和每页数量未限制合法范围，异常请求可能放大数据库查询和响应体开销。

### 2.5 首页数据可能长期陈旧

经营概览只在首次进入和手动点击时刷新，长时间打开页面后无法及时反映新订单和库存预警。

### 2.6 商品资料缺少维护入口

商品管理当前只有查看、同步、配置和本地删除，标题、价格、描述与封面等本地运营资料无法直接维护。现有平台连接只具备商品读取能力，没有可验证的商品写入授权，不能把本地保存伪装为闲鱼商品已更新。

## 3. 状态与操作设计

| 履约状态 | 页面文案 | 用户操作 | 自动重试 |
| --- | --- | --- | --- |
| `PENDING` | 等待处理 | 查看详情 | 由任务调度处理 |
| `PROCESSING` | 正在处理 | 查看详情 | 不允许重复触发 |
| `RETRY_WAIT` | 等待重试 | 查看详情 | 由任务调度处理 |
| `DELIVERED` | 已交付 | 查看详情、确认发货 | 无需 |
| `CONFIRMING` | 确认中 | 查看详情 | 由任务调度处理 |
| `REVIEW_REQUIRED` | 需要人工核对 | 查看本地或平台详情、人工确认 | 禁止 |
| `FAILED` | 履约失败 | 查看原因、重新排队 | 允许人工触发 |
| `COMPLETED` | 已完成 | 查看详情、确认发货 | 无需 |
| `SKIPPED` | 已跳过 | 查看详情 | 无需 |

安全边界：

- 重新排队接口只接受当前状态为 `FAILED` 且 `state <> 1` 的订单。
- `REVIEW_REQUIRED` 表示发送结果不确定，禁止自动或人工一键重发。
- `PROCESSING` 和 `RETRY_WAIT` 已进入任务调度流程，不提供重复操作按钮。
- 重新排队使用数据库条件更新保证并发幂等，状态已变化时返回明确提示。

## 4. 页面设计

### 4.1 经营概览

待办跳转规则：

- 等待履约：进入发货记录并筛选 `PENDING,PROCESSING,RETRY_WAIT`。
- 需要人工核对：进入发货记录并筛选 `REVIEW_REQUIRED`。
- 履约失败：进入发货记录并筛选 `FAILED`。
- 卡密库存预警：进入卡密管理并仅展示低库存仓库。

页面在浏览器可见时每 60 秒刷新一次，页面进入后台时停止刷新，重新可见时立即刷新。该频率兼顾待办时效和数据库占用。

### 4.2 发货记录

在现有关键词筛选旁增加履约状态筛选，桌面端和移动端共用同一查询条件。

状态展示优先使用 `deliveryStatus`，兼容旧数据时回退到 `state`。失败原因优先使用业务字段 `failReason`，为空时回退到任务字段 `lastErrorMessage`。

仅 `FAILED` 状态显示“重新排队”按钮。操作前使用现有确认弹窗，成功后刷新当前列表。

从首页进入时读取路由查询条件并立即筛选；重置操作同时清除关键词、履约状态和路由筛选，避免页面状态与地址不一致。

### 4.3 卡密管理

从首页库存预警进入时，根据现有仓库的预警开关、阈值类型、阈值数值和可用数量筛选低库存仓库。

低库存判断保持与后端统计一致：

- 数量阈值：可用数量小于阈值。
- 百分比阈值：总量大于零，且可用比例小于阈值。
- 未启用预警的仓库不进入低库存视图。

页面保留返回全部仓库的筛选入口，避免低库存筛选造成数据隐藏误解。

### 4.4 商品资料编辑

商品列表在桌面端和移动端增加“编辑”入口，弹窗只保留标题、价格、商品描述和封面地址四个常用字段。保存按钮明确标注为“保存本地资料”，成功后刷新当前列表。

平台状态、SKU、库存、图集和分类继续只读，避免修改本地镜像后破坏卡密规格匹配或造成平台数据不一致。弹窗同时提供“前往闲鱼修改”入口，优先使用现有商品详情地址，地址缺失时根据商品 ID 打开商品页；完成平台修改后复用现有单商品同步能力回读最新资料。

弹窗展示简短说明：本地资料用于系统展示与运营配置，后续同步会以闲鱼平台数据为准。界面不增加多层向导或装饰性组件。

## 5. 接口设计

### 5.1 发货记录查询

请求增加：

```json
{
  "deliveryStatuses": ["FAILED"]
}
```

后端只接受 `DeliveryStatus` 枚举中存在的值，无效状态返回参数错误。`pageNum` 最小为 1，`pageSize` 限制在 1 至 100。

响应增加：

```json
{
  "deliveryStatus": "FAILED",
  "failReason": "连接不可用"
}
```

查询列表与总数必须使用完全一致的筛选条件。

### 5.2 失败任务重新排队

新增接口：

```text
POST /api/order/requeueDelivery
```

请求包含订单记录 ID 和账号 ID。服务端校验记录存在、账号归属、当前履约状态和传统发货状态，满足安全条件后将任务更新为 `PENDING` 并立即进入调度范围。

数据库更新条件包含账号、记录 ID、`state <> 1` 和 `delivery_status = 'FAILED'`，受影响行数为零时返回“订单状态已变化，请刷新后重试”。

### 5.3 本地商品资料更新

新增接口：

```text
POST /api/items/updateInfo
```

请求包含账号 ID、商品 ID、标题、价格、商品描述和封面地址。服务端使用账号 ID 与商品 ID 联合定位记录，并校验标题非空且不超过 256 字、价格非负且最多两位小数、描述不超过 5000 字、封面地址为空或使用 HTTP/HTTPS 协议。

更新语句只写入上述四个字段和更新时间，不修改平台状态、SKU、库存、图集、详情地址或自动化配置。账号归属不匹配时统一返回商品不存在，避免跨账号更新。

## 6. 数据库与性能

新增 Flyway 迁移，为发货记录的账号、履约状态和创建时间查询增加组合索引：

```sql
CREATE INDEX idx_goods_order_account_delivery_created
    ON xianyu_goods_order (xianyu_account_id, delivery_status, create_time);
```

保留现有任务调度索引，两个索引分别服务全局任务领取和单账号后台查询，不替换现有索引。

首页自动刷新复用现有聚合接口，不新增轮询线程或服务端推送组件。

## 7. 异常处理

- 列表查询失败沿用统一请求层提示，不重复弹出相同错误。
- 重新排队失败保留当前列表状态，并展示服务端返回原因。
- 路由包含非法状态时忽略非法值并回退为全部状态。
- 低库存筛选没有结果时展示明确空状态，并保留查看全部仓库入口。
- 自动刷新失败不清空已有经营数据，下一周期继续刷新。
- 商品编辑校验失败时保留输入内容；保存成功后关闭弹窗并刷新列表。
- 平台详情地址无效时不执行跳转，并显示明确提示。

## 8. 验收标准

### 8.1 用户体验

1. 首页四类待办均可一步到达对应结果。
2. 发货记录可区分全部履约状态并展示准确失败原因。
3. 明确失败订单可确认后重新排队，人工核对订单不显示重试按钮。
4. 桌面端和移动端均可筛选状态。
5. 低库存入口只展示触发预警的仓库，并可恢复全部仓库。
6. 页面长期打开时数据保持更新，后台标签页不产生周期请求。
7. 商品列表可直接编辑标题、价格、描述和封面等本地资料。
8. 平台状态、SKU 与库存不会在本地编辑中被误改。
9. 可从编辑弹窗前往闲鱼修改，返回后使用现有同步按钮回读平台资料。

### 8.2 开发与性能

1. 后端编译和现有测试通过。
2. 前端类型检查与生产构建通过。
3. 查询页码和每页数量边界生效。
4. 查询列表与总数使用相同状态条件。
5. 重新排队更新具备账号和状态条件，不允许已完成或待核对订单进入队列。
6. Flyway 迁移可在现有数据库顺序执行。
7. 本轮新增的临时测试代码和测试数据在验证完成后删除。
8. 商品资料更新具备账号归属和字段边界校验，更新范围不包含履约相关字段。

## 9. 范围边界

本轮不拆分大型服务类和页面，不引入通知中心、消息队列、Redis、批量重试或新的视觉组件库，不调整 Cookie、WebSocket、自动回复和实际卡密发送流程，不接入未经授权的平台写接口，不开放 SKU、库存和平台状态编辑，不迭代 Release。
